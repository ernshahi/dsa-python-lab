## System Design Interview Checklist

Use this checklist in every design round so your answer stays structured and complete.

### 1. Clarify Requirements
- Who are the users?
- What are the main use cases?
- What is in scope and out of scope?
- What scale should be assumed: requests per second, users, storage, regions?
- What are the latency, availability, durability, and consistency expectations?

### 2. Define APIs and Data Flow
- State the main read and write APIs
- Describe the request flow end to end
- Identify synchronous vs asynchronous paths

### 3. Estimate Scale
- Estimate QPS, peak traffic, daily active users, and storage growth
- Estimate read/write ratio
- Note hot keys, hot partitions, and burst traffic patterns

### 4. Propose High-Level Architecture
- Clients
- Load balancer
- Application services
- Database
- Cache
- Queue or stream
- Blob or file storage
- Search index if needed

### 5. Choose Data Storage Carefully
- SQL vs NoSQL
- Primary access patterns
- Indexing strategy
- Partitioning strategy
- Replication strategy
- Backup and recovery basics

### 6. Address Scalability
- Horizontal scaling plan
- Stateless service design
- Caching strategy
- Read replicas
- CDN where relevant
- Queue-based buffering for spikes

### 7. Discuss Reliability
- Single points of failure
- Failover strategy
- Retry and timeout policy
- Idempotency
- Disaster recovery
- Multi-AZ or multi-region approach if relevant

### 8. Discuss Consistency and Tradeoffs
- Strong vs eventual consistency
- Latency vs correctness tradeoffs
- Cost vs performance tradeoffs
- Simplicity vs extensibility tradeoffs

### 9. Cover Security and Privacy
- Authentication and authorization
- Encryption in transit and at rest
- Rate limiting and abuse prevention
- PII handling and auditability

### 10. Cover Observability and Operations
- Logs
- Metrics
- Tracing
- Alerting
- Capacity planning
- Operational runbooks

### 11. Deep Dive on One or Two Critical Components
- Data model
- Cache invalidation
- Ranking logic
- Queue processing
- Notification fan-out
- Search indexing

### 12. End With Risks and Future Improvements
- State the biggest bottlenecks
- Explain what you would improve next
- Mention how the design changes at 10x scale

### Quick Delivery Template
- Requirements
- APIs
- Estimations
- High-level design
- Database and data model
- Scaling
- Reliability
- Tradeoffs
- Deep dive
- Risks and next steps
